## Ticket 0 — Vocabulary + slug files (prerequisite for everything else)

**Why:** there is currently no stable slug for agents, and disease slugs exist only as filenames
with at least one confirmed duplicate/near-duplicate (see Known Data Issues below). Every later
ticket needs a slug it can trust not to change.

**Do:**
1. Create `data/vocab/disease-slugs.yaml`. Enumerate every file in `data/disease-intelligence/*.json`.
   For each, record `{slug: <filename stem>, mondo_id: <identifiers.mondo_id or null>, canonical_label: <condition or page.title>}`.
   Where two files clearly refer to the same condition (e.g. `post-acute-covidvaccination-syndrome`
   vs `post-acute-covid-vaccination-syndrome` — confirmed duplicate pair, one is a 1.6KB stub), pick
   the larger/richer file as canonical and record the other as `{slug: ..., alias_of: <canonical slug>}`.
   Do not delete the stub file yet — just record the decision.
2. Create `data/vocab/agent-slugs.json`. Read `data/therapeutic_agents.json` (`.agents[]`, keyed by
   the literal field `"Therapeutic Agent"` — there is no existing slug or ChEBI/RxNorm id, it's a
   free-text name, sometimes a full sentence like `"140 Ml Per Day Of Beet-It Nitrate Beverage"`).
   Slugify: lowercase, strip punctuation, hyphenate, collapse whitespace. Detect collisions after
   slugification and disambiguate with a numeric suffix. Output one record per agent:
   `{slug, display_name: original "Therapeutic Agent" value}`. Flag (log, don't fix) any agent name
   that reads as a dosing instruction rather than an agent name (e.g. contains a number + unit) —
   these are pipeline extraction artifacts, not real agents, and should NOT get their own Template C
   page later; list them in `data/vocab/agent-slugs-flagged.json` instead of the main file.
3. Create `data/vocab/cohort-slugs.json` from `data/cohorts/*.json` — the `id` field already exists
   and is filename-stable; just add a `short_slug` (kebab-case of `name`, truncated to a
   reasonable length) so Template D URLs can be `/cohorts/<id>-<short_slug>/`.

**Gate:** all three vocab files exist, are valid JSON/YAML, `disease-slugs.yaml` has one entry per
disease-intelligence file with no two non-alias entries sharing a `mondo_id`, `agent-slugs.json` has
no duplicate slugs.

**Do NOT physically move `disease-intelligence/` or `pais-cohorts/` yet** — that's a large,

1. Write the decision to `data/vocab/url-map.yaml`: for each content type, record
   `{current_path_pattern, target_path_pattern, migrate: true|false, reason}`. Recommendation to

2. Add a single redirect: `pais-cohorts.html` → `/cohorts/` only once Ticket 3 (below) actually

**Gate:** `data/vocab/url-map.yaml` exists and states explicitly which trees move and which don't,

## Ticket 2 — Thin-content gate as a reusable script (build once, use everywhere)

**Why:** strategy doc §4 is the whole point of this project (mass thin pages tank the whole
domain). It must be enforced by a script, not eyeballed per template.

**Do:** write `scripts/lib/thin-content-gate.py` (or `.js`, match the generator that calls it) with
one function `gate(page_text, min_words, corpus_dir) -> {indexable: bool, reasons: []}` that checks:
1. Word count of the page's unique prose block (the human-authored summary, not boilerplate
   nav/footer/citation-list markup) ≥ `min_words` (250 for Template A analog, 200 for B, 150 for D
   — pass as a parameter, don't hardcode).
2. 5-gram shingle overlap against every other already-built page's prose ≤ 70% (MinHash or brute
   set overlap over a small corpus is fine — corpus here is at most a few hundred pages initially,
   no need for approximate MinHash at this scale; exact Jaccard over 5-gram sets is fine and
   simpler to debug).
3. At least one citation with a resolvable PMID or DOI string is present in the page's citation
   list.
Return `noindex,follow` meta-robots content when any check fails, `index,follow` when all pass.
Also write a `--report` mode that prints a table (page, pass/fail per rule) so a human/agent can
see the noindex backlog at a glance.

**Gate:** running the script against 5 already-existing `disease-intelligence/*.html` pages (as a
smoke test — read their existing prose, don't regenerate anything) produces a report with no
crashes and plausible word counts.

## Ticket 3 — Template D: cohort pages get Dataset JSON-LD + gating (smallest, ship first)

**Why:** cohorts are the richest, most complete data (schema v2.0.0, `data/cohorts/*.json`) and the
strategy doc calls out `Dataset` JSON-LD as the highest-value, lowest-competition win (Google
Dataset Search). This is the correct first template to ship per the strategy doc's build order.

**Real field names to use** (confirmed from `data/cohorts/*.json`, schema
`data/pais-cohort.schema.json`) — do not use the strategy doc's assumed names:
- `pathogen_id` (not `pathogen`) — resolve display name via `data/ref/pathogens.json`.
- No `country` field — check `recruitment_source` / free text in `notes`; if no structured
  location exists, omit `spatialCoverage` from JSON-LD rather than inventing one.
- No `year_range` field — derive `temporalCoverage` from `publications[].year` (min–max across
  `publications[]`), falling back to `outbreak_event` if present.
- `n_analysed` (not `n_followed`); `followups[]` for follow-up duration detail.
- Primary citation: filter `publications[]` for `is_primary_cohort_paper: true`, use its
  `doi`/`pmid`.
- Biospecimen availability: `biobank_status` + `external_access` (not a single
  `biospecimen_availability` field).
- Outcomes measured: derive from `observations[].measure_id` (resolve display names via
  `data/ref/measures.json`), not a top-level `outcomes_measured[]` array.

**Do:**
1. Extend `scripts/build_pais_cohorts.py` (do not write a parallel generator — this script already
   builds `pais-cohorts/<id>.html` correctly from the same source data) to also emit, per cohort
   page, a `<script type="application/ld+json">` block of `@type: Dataset` using the field mapping
   above. Emit `null`/omit optional JSON-LD keys rather than fabricating values for fields that
   don't exist on a given record.
2. Call `thin-content-gate.py` (Ticket 2) per cohort page with `min_words=150`; if a cohort fails,
   set `<meta name="robots" content="noindex,follow">` in that page's `<head>` and record it in a
   `content-debt.json` log (page, reasons) instead of adding it to the sitemap list.
3. Also emit one corpus-level `Dataset` JSON-LD block at `data/pais-cohorts-index.json`'s
   corresponding HTML page (`pais-cohorts.html` itself), describing the cohort corpus as a whole.
4. Add `pais-cohorts` (already present) confirmed in `scripts/build_site_seo.py`'s `PUBLIC_DIRS` —
   check it's there; if not, add it, otherwise this ticket's pages won't reach the sitemap.

**Do NOT** in this ticket: register with Google Dataset Search / re3data / DataCite, mint a Zenodo
DOI — those are manual account-setup actions for a human, not something a coding agent can do; flag
them as a follow-up TODO in the PR description instead.

**Gate:** re-run `scripts/build_pais_cohorts.py`; every emitted cohort page has valid JSON-LD
(validate with a local schema.org shape check — at minimum required keys `@context`, `@type`,
`name`, `description`); `content-debt.json` lists any cohorts below the 150-word/citation bar;
`sitemap.xml` (regenerate via `scripts/build_site_seo.py`) contains only the non-flagged cohort
pages.

## Ticket 4 — Template A: retrofit existing disease-intelligence pages with real thin-content gate + missing fields

**Why:** Template A pages already exist and are pre-rendered (`disease-intelligence/<slug>.html`,
~135 of them) with real SEO metadata already in the data (`page.title`, `page.description`,
`page.canonical`, `page.keywords[]`). The strategy doc's Template A is **not a new page type here
— it's an existing one that needs the missing fields backfilled and the thin-content gate applied
before scaling.** The strategy's Phase 3 gate ("ship 20 pages only, fully reviewed") should be
read as: pick 20 of the existing 135 pages, backfill their missing fields, gate them, and treat the
rest as still-unreviewed content debt.

**Fields the strategy doc assumes that do not exist yet — these are net-new additions, not joins:**
- `prevalence_estimate {value, unit, source_pmid}` — does not exist anywhere. Do not fabricate; only
  add where you can cite a real source (GBD burden data may already be in `disease_pipeline/` per
  `build_gbd_burden.py` — check `data/disease-intelligence/<slug>.json:burden` before treating this
  as fully missing).
- `guideline_last_updated {year, body, url}` — does not exist. Leave null unless a real source is
  found; do not invent a plausible-sounding year.
- `approved_treatments[] {agent, indication, approval_year, agency}` — closest existing analog is
  `therapeutics.direct[]` filtered for `max_phase >= 4` (FDA-approval proxy) — this is an
  **approximation**, not a verified `approved_treatments` list; label it as such in any prose
  ("phase-4 candidates" not "FDA-approved") until someone verifies agency + approval year per drug.
- `candidate_agents[] >= 3 {agent_slug, evidence_grade, n_studies}` — use
  `therapeutics.merged_ranked[]` (already has `canonical_id`, `name`, `evidence_tier`,
  `pubmed_count` — map `pubmed_count`→`n_studies`, `evidence_tier`→`evidence_grade`,
  `canonical_id`→`agent_slug` via the Ticket 0 agent-slug map by matching on name, not by assuming
  `canonical_id` is already a slug).

**Do:**
1. Pick 20 disease-intelligence pages with the richest existing `therapeutics.merged_ranked[]`
   (≥3 entries — this is the hard requirement from spec §4.2) as the Phase-3 sample.
2. For each, run `thin-content-gate.py` with `min_words=250` against the existing `summary` +
   `natural_review`/`clinical_evidence` prose blocks.
3. Add `MedicalCondition` JSON-LD (per strategy doc §6 shape) using `identifiers.mondo_id`,
   `condition`/`page.title`, and citations pulled from `clinical_evidence` — this is new code in
   whatever script renders `disease-intelligence/<slug>.html` (find it via `disease_pipeline/export_web.py`
   or `publish_site.py`; do not hand-edit the generated HTML files directly).
4. Do not touch the other ~115 pages in this ticket — they stay as-is, un-gated, until Phase 4.

**Gate:** 20 pages identified, each with a JSON-LD `MedicalCondition` block, each passing or
explicitly failing (and noindexed) the thin-content gate, `approved_treatments`/`candidate_agents`
derivations clearly labeled as approximations in the data (not silently presented as verified). Per
strategy doc §11: do not proceed to Template A at full scale until ≥8 of these 20 rank top-50 for a
target query at 8 weeks post-ship — that check is time-gated and happens outside this repo (GSC),
not something to simulate.

## Ticket 6 — Template B: disease × agent pages (the volume play — do last, per strategy doc's own ordering)

**Why:** strategy doc explicitly orders this after A and C are stable (`Phase 5` in its build
order) because it's the highest-volume, highest-risk-of-thin-content template. Do not start this
before Tickets 3–5 have shipped and the Template A 20-page cohort (Ticket 4) has had time to be
evaluated — that evaluation is an external, time-gated check (GSC ranking data), not something this
ticket can verify itself.

**Data join:** a (disease, agent) pair page requires, per strategy doc §3, `evidence_grade`,
`studies[] >= 1`, `mechanism_rationale`, `safety_notes`. The actual source for this join is
`data/disease-intelligence/<slug>.json:therapeutics.merged_ranked[]` (per-disease, has the agent
name/id and evidence tier) crossed with `data/therapeutic_agents.json` (per-agent mechanism/safety,
keyed by matching name — there is no shared numeric id between the two files, so the join is a
name-normalization problem: lowercase + strip punctuation + fuzzy-match, and any pair that doesn't
match confidently should be skipped, not force-joined).

**Do:**
1. Build the join table first as a standalone script `scripts/build_disease_agent_pairs.py` that
   outputs `data/disease-agent-pairs.json` (list of `{disease_slug, agent_slug, evidence_tier,
   n_studies, match_confidence}`) — inspect `match_confidence` manually for a sample before trusting
   it at scale.
2. Only generate an actual HTML page for pairs with `n_studies >= 1` (hard requirement) — most
   pairs from `merged_ranked[]` likely already satisfy this since they're derived from
   pubmed-backed scoring; verify rather than assume.
3. Prose (`mechanism_rationale`, `safety_notes`) must be genuinely generated per pair, not templated
   sentence substitution ("Evidence for {agent} in {disease} is {grade}") — strategy doc §4 is
   explicit that this is exactly what gets penalized as scaled content abuse. If prose generation
   requires an LLM pass, store the generated text as a reviewable file in the repo
   (`data/generated-prose/disease-agent/<disease_slug>__<agent_slug>.json` with the text plus
   `{generated_by, generated_at}` front matter) rather than generating it at render time.
4. Apply the thin-content gate at `min_words=200` before any page ships with `index,follow`.
5. Ship in small batches (tens, not hundreds) and check `scripts/lib/thin-content-gate.py --report`
   after each batch — the strategy doc's kill criteria (§12: flat clicks despite growing indexed
   count, GSC "Crawled – currently not indexed" > 30%) apply here hardest since this is the biggest
   page count.

**Gate:** join table reviewed for match_confidence quality on a sample; first batch ≤ 50 pages;
every shipped page passes the gate; none use template-substitution prose (spot check the generated
text for repeated sentence skeletons across pages — if you can spot the pattern by eye, so can
Google).

## Ticket 7 — Internal-link CI check (cross-cutting, do once Tickets 3–6 have produced enough pages to test against)

**Why:** strategy doc §7: orphan pages don't rank; this must be enforced by CI, not by hand-curation.

**Do:** write `scripts/check_internal_links.py` that:
1. Crawls the generated output directories (`pais-cohorts/`, `disease-intelligence/`, `agents/`,
   whatever Template B lands in) and builds a link graph from `<a href>` tags.
2. Fails (non-zero exit) if any indexable page (i.e. not `noindex`-flagged per the thin-content
   gate) has fewer than 3 distinct inbound links from other pages in the graph.
3. Fails if any indexable page has fewer than 3 outbound internal links.
Wire this into whatever CI runner the repo uses (check for an existing `.github/workflows/` — if
none exists, just make this a script the human runs before deploy; don't invent a new CI system as
part of an SEO ticket).

**Gate:** running the script against the current output of Tickets 3–6 produces a list of orphan
pages (expect some on first run — that's the point); zero orphans among pages marked `index,follow`
before this is considered done.
